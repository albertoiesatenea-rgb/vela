Quiero una optimización seria de tokens, coste y latencia de mi app, basada en el código ACTUAL, sin cambiar la funcionalidad visible, la potencia táctica ni la operativa del producto.

IMPORTANTE
No quiero rediseño.
No quiero simplificaciones funcionales.
No quiero perder calidad.
No quiero “ahorrar tokens” a costa de volver la herramienta más tonta o menos útil.

El objetivo es:
- mismo producto por fuera
- menos gasto por dentro
- menos redundancia
- menos payload inútil
- misma o mejor robustez

CONTEXTO TÉCNICO ACTUAL QUE DEBES TOMAR COMO VERDAD
Ya he inspeccionado la app actual y estos son los hallazgos clave:

1. COPILOTO
- /api/copilot/analyze usa gpt-4o-mini con max_tokens=900
- envía en cada turno un BASE_SYSTEM_PROMPT de ~2.100 tokens fijos
- ese prompt incluye ejemplos completos que suman ~400 tokens
- se reenvía entero en CADA turno
- además manda contexto de sesión variable + call_memory comprimida + fragmento actual
- no hay historial completo, solo call_memory

2. ARENA
- /api/arena/turn usa gpt-4o-mini con max_tokens=300
- en cada turno se reenvía el historial completo de la conversación como mensajes nativos GPT
- el historial crece sin truncado ni compresión
- además, en seller mode, se hace otra llamada extra detectTerminalState en cada turno
- /api/arena/suggest también manda transcript completo
- /api/arena/finish manda transcript completo para debrief

3. MÉTRICAS
- hoy no se registran usage.prompt_tokens ni usage.completion_tokens
- no existe coste por llamada, por sesión, por ruta ni por modo

4. ESTADO
- Copiloto ya usa una call_memory compacta
- Arena no usa memory pack ni resumen, solo historial completo creciente

OBJETIVO DE ESTA TAREA
Optimizar internamente la app para reducir tokens y llamadas innecesarias SIN cambiar:
- pantallas
- flujos
- nombres visibles
- schema de salida de Copiloto
- comportamiento esperado de Arena
- utilidad táctica del producto

REGLA ABSOLUTA
Si una optimización pone en riesgo calidad o comportamiento, no la fuerces sin red de seguridad.
Quiero una implementación segura:
- con validación
- con fallback
- con cambios limpios

ESTRATEGIA QUE QUIERO QUE SIGAS

FASE 1 — INSTRUMENTACIÓN Y BASE SEGURA
Antes de optimizar, añade visibilidad real del consumo:

Además, añade una forma simple de activar/desactivar optimizaciones mediante flags internas si hace falta.
Quiero poder mantener un camino legacy por seguridad si algún cambio sale mal.

FASE 2 — OPTIMIZACIÓN DE COPILOTO
Optimiza /api/copilot/analyze sin cambiar el output visible ni el schema.

Objetivos concretos:
1. Reducir drásticamente el tamaño del BASE_SYSTEM_PROMPT
2. Mantener toda la lógica táctica importante
3. Mantener mismo schema JSON final
4. Mantener utilidad táctica igual o mejor

Qué quiero que revises y optimices:
- eliminar redundancias del prompt base
- comprimir definiciones que hoy están explicadas de forma larga
- reducir o eliminar ejemplos extensos si no aportan valor real
- evitar repetir explicaciones obvias del schema si se puede imponer de otra forma
- compactar instrucciones sin perder reglas críticas
- unificar reglas duplicadas
- mantener reglas esenciales:
  - anti-repetición
  - diferencia entre signal / reading / mission / journey.now / say_now
  - reglas de support
  - cierre / no cierre prematuro
  - call_memory útil
  - momentum

Además:
- preconstruir y cachear por sesión el bloque de contexto formateado para no recomponerlo inútilmente en cada turno
- cachear serialización de memory pack si tiene sentido
- no enviar texto redundante si ya puede ir compacto

IMPORTANTE:
No quiero una simple “cache de strings” que no ahorre tokens reales.
Quiero ahorro real en payload al modelo.

FASE 3 — OPTIMIZACIÓN DE ARENA
Optimiza Arena sin cambiar la experiencia visible de conversación.

Objetivos:
1. dejar de reenviar historial completo sin límite
2. mantener coherencia conversacional
3. mantener personalidad / contexto / rol
4. reducir latencia y tokens

Qué quiero que implementes:
- introducir una memoria compacta de Arena por sesión, tipo arenaMemory o summaryPack
- usar un contexto híbrido:
  - resumen compacto acumulado
  - últimos N turnos recientes
- no enviar siempre todo el historial completo si no hace falta
- elegir un windowing conservador y seguro

REGLA DE SEGURIDAD
Haz esta optimización de forma prudente.
No quiero que Arena “olvide” lo importante.
Prefiero un ahorro sólido y conservador antes que una compresión agresiva.

Además:
- revisa las descripciones largas de perfil/dificultad dentro del system prompt de Arena y compáctalas
- revisa /arena/suggest para que no mande transcript completo si no hace falta
- revisa /arena/finish para que el debrief use una fuente eficiente sin perder calidad
- revisa detectTerminalState

OPTIMIZACIÓN ESPECÍFICA DE detectTerminalState
Hoy hay una llamada adicional en seller mode en cada turno.
Quiero que estudies si se puede ahorrar esa llamada sin perder funcionalidad.

Opciones válidas, elige la más limpia:
- integrar terminal state en la misma llamada principal de arena/turn
- o ejecutar la detección solo cuando tenga sentido real
- o alguna alternativa equivalente

Pero:
- no quiero perder la capacidad actual de detectar cierre / next_step / lost / broken
- no quiero añadir una lógica frágil
