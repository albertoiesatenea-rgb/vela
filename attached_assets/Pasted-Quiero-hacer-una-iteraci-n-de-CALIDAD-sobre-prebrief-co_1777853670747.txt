1) MEJORAR `prebrief-context` PARA LEER CASOS ESTRUCTURALES
==================================================

Quiero que `POST /api/copilot/prebrief-context` deje de priorizar objeciones tópicas del negocio cuando el input contiene señales más específicas del caso.

Nueva jerarquía de lectura obligatoria del input:
1. evento explícito de la llamada
2. hechos estructurales del caso
3. restricciones reales de avance
4. stage / labels del CRM
5. objeciones típicas del negocio solo si no hay señales más fuertes

Reglas nuevas obligatorias del prompt:
- No uses objeciones típicas del sector como “cashflow negativo”, “desconocer Alemania” o “tipos altos” por defecto si el input contiene fricciones más específicas.
- Si el input contiene detalles fiscales, laborales, geográficos, familiares, de permanencia, financiación o scoring, promuévelos por encima de objeciones genéricas.
- Si el caso tiene complejidad transfronteriza, horizonte de permanencia corto, score dudoso, estructura familiar relevante o restricciones de financiación, eso debe pesar en la lectura del caso.
- `main_blocker_probable` debe ser un único freno dominante, no una lista.
- `today_decision` debe reflejar la decisión comercial real de esta fase, no un resumen administrativo.
- `what_client_knows` sigue significando solo lo que el cliente ya conoce del proceso/modelo/llamada, NO datos personales o financieros.
- `context_for_brief` debe arrastrar explícitamente las restricciones y riesgos estructurales del caso.

2) AÑADIR 3 CAMPOS NUEVOS AL OUTPUT DE `prebrief-context`
==================================================

Mantén los campos actuales, pero añade estos 3 campos nuevos al JSON de salida:

{
  "special_context_flags": ["string"],
  "decision_constraints": ["string"],
  "case_specific_risks": ["string"]
}

Definición exacta:

- `special_context_flags`
  = señales estructurales del caso que cambian la lectura táctica
  Ejemplos:
  - "situación fiscal/laboral transfronteriza"
  - "permanencia en Alemania probablemente limitada"
  - "estructura familiar relevante"
  - "score crediticio no limpio"
  - "geografía objetivo muy concreta"

- `decision_constraints`
  = cosas que condicionan lo que se puede decidir hoy
  Ejemplos:
  - "no empujar propuesta sin validar encaje España/Alemania"
  - "necesita confirmar si su caso es financiable en estas condiciones"
  - "conviene validar permanencia antes de pasar a propuesta"

- `case_specific_risks`
  = riesgos de lectura o de avance específicos de ESTE caso
  Ejemplos:
  - "leer el caso como objeción genérica de rentabilidad cuando el freno real es de encaje"
  - "pasar a propuesta demasiado pronto"
  - "confundir interés real con encaje real"

Importante:
- máximo 3 ítems por campo
- compactos
- específicos
- nada de relleno
