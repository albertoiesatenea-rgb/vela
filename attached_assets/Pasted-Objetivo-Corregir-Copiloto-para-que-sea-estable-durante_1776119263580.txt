Objetivo
 Corregir Copiloto para que sea estable durante toda la llamada y, además, elevar la calidad comercial del guidance cuando la objeción del cliente es técnica y concreta. El parche anterior mejoró transcript, debrief_reliable y honestidad del resumen, pero el analyze sigue fallando a ratos y el cerebro táctico sigue empujando a reencuadres demasiado abstractos.
Contexto del problema
 He probado otra llamada de Copiloto con el caso José Romero tras el parche anterior. El sistema ahora muestra mejoras visibles:
transcript final visible
banner de “Debrief no fiable”
score ya no inflado absurdamente
model metadata aparece como gpt-4o en el audit log
Pero siguen dos fallos graves:
Inestabilidad real de Copiloto
 En el audit log aparecen 4 turnos con response_status: error y parse_error: API runtime error (turnos 3, 4, 11 y 14). Esto coincide con el comportamiento observado en UI: tras hablar el cliente, a veces aparece “actualizando” y luego la pantalla se queda exactamente igual, con guidance vieja. Esto significa que el frontend ya informa mejor del problema, pero el runtime sigue cayéndose demasiado para una sesión real.
Guidance comercial insuficientemente fino
 Aunque algunas lecturas son buenas, varios say_now / mission / next_move siguen siendo demasiado abstractos o tardíos:
“confirma renta baja, pivota a seguridad futura”
“pregunta qué nivel de estabilidad busca”
 En este caso el cliente ya había concedido que valora la estabilidad. La objeción real era técnica: renta demasiado baja respecto al precio y falta de defendibilidad del alquiler actual. El guidance debía empujar a:
responder con precisión qué se sabe y qué no se sabe sobre subida de renta
distinguir contrato antiguo vs. posibilidad real de actualización
usar la garantía de 6 años sin vender humo
y decidir si el activo sigue siendo defendible o no
 No debía volver a generalidades de “estabilidad vs rentabilidad”.
Además, el debrief sigue siendo algo generoso:
resultado: “Siguiente paso acordado”
 cuando en realidad el cliente acaba yéndose con un “pásame otra propuesta por correo y si tengo tiempo la miro”, lo cual no es un siguiente paso fuerte sino un arrastre débil.

La última prueba de Copiloto no se hizo en modo escuchar ni dependía de transcripción de audio. Se hizo en modo escribir/simular hablantes, alternando manualmente YO y CLIENTE. El audit log ya refleja source_mode: simulate, speaker_mode_default: client, speaker_mode por turno y normalized_input con etiquetas [YO] y [CLIENTE]. Aun así hubo varios API runtime error, guidance congelada y actualizaciones incompletas de pantalla. Por tanto, este fallo no debe tratarse como problema de STT/micrófono, sino como problema real de analyze/runtime/guidance.
